# Roadmap: GMX Audit Control Center

## 1. Delivery strategy
- Ship docs and schema contracts first.
- Deliver a visible static dashboard MVP early.
- Expand iteratively through findings/proof detail depth.
- Keep managed operations stable while improving observability.

## 2. MVP definition
MVP includes:
- Rebranded README and Control Center docs.
- Schema contracts for run/finding/proof/score/manifest.
- Static overview dashboard with trend cards.
- Findings explorer and proof package pages.
- Managed-mode baseline visibility.

## 3. Week-by-week execution plan

## Week 1: Foundation and contracts
Objective: lock product framing and data contracts.

Epics:
- Epic CC-01: Product and messaging baseline
- Epic CC-02: Data contract baseline

Issues:
1. CC-1 Rewrite README for Control Center positioning
   - Acceptance: README is public-facing, scoped, and claim-safe.
2. CC-2 Add docs/prd-control-center.md with P0/P1/P2 priorities
   - Acceptance: PRD approved by maintainers.
3. CC-3 Add schema files for run/finding/proof/score/manifest/triage
   - Acceptance: schema validation script passes on sample outputs.
4. CC-4 Add docs/messaging-guardrails.md
   - Acceptance: approved/avoid phrases documented.
5. CC-5 Add examples/sample-dashboard static artifact
   - Acceptance: sample page viewable locally without build tools.

Dependencies:
- Existing output format availability.

Risks:
- Schema scope creep.

Definition of done:
- README + PRD + schema contracts merged.

## Week 2: Dashboard foundation
Objective: deliver first Control Center overview page from real outputs.

Epics:
- Epic CC-03: Static overview dashboard

Issues:
1. CC-6 Create dashboard template shell and layout
   - Acceptance: renders cards/charts/table placeholders.
2. CC-7 Wire triage-result ingestion adapter
   - Acceptance: finding count and severity cards populate.
3. CC-8 Wire SQLite trend adapter
   - Acceptance: score/run trend chart renders from history.
4. CC-9 Implement static exporter with no-CDN assets
   - Acceptance: outputs/metrics/dashboard.html generated in CI.
5. CC-10 Add empty/error/loading states
   - Acceptance: states visible for missing/partial data.

Dependencies:
- Week 1 schema and docs complete.

Risks:
- Data shape drift between historical runs.

Definition of done:
- Control Center overview generated by one command and visible in CI artifacts.

## Week 3: Findings and proof navigation
Objective: make triage and packaging workflows navigable end-to-end.

Epics:
- Epic CC-04: Findings explorer
- Epic CC-05: Proof package views

Issues:
1. CC-11 Build findings index page with filters
   - Acceptance: filter by severity/status/chain works.
2. CC-12 Build finding detail page with evidence refs
   - Acceptance: links to run/proof/artifacts resolve.
3. CC-13 Build proof package detail page
   - Acceptance: lists proof.json, summary, repro scripts, report.
4. CC-14 Add manifest link validation in build pipeline
   - Acceptance: CI fails on missing artifact references.
5. CC-15 Add run-to-finding and finding-to-proof cross-links
   - Acceptance: user can navigate full path in two clicks.

Dependencies:
- Week 2 exporter and adapters.

Risks:
- Broken links from retention deletions.

Definition of done:
- Triager can open finding and proof pages directly from dashboard.

## Week 4: Score explainability and managed visibility
Objective: improve trust, operations clarity, and launch readiness.

Epics:
- Epic CC-06: Score explainability
- Epic CC-07: Managed operations surface

Issues:
1. CC-16 Add score explanation panel and version metadata
   - Acceptance: score formula components are visible.
2. CC-17 Add finding delta view between latest two runs
   - Acceptance: new/regressed/resolved counts displayed.
3. CC-18 Expose managed scheduler status summary
   - Acceptance: run lock/retry/retention status visible per client.
4. CC-19 Publish docs/architecture-control-center.md migration map
   - Acceptance: current-to-target mapping approved.
5. CC-20 Publish docs/launch-positioning-control-center.md
   - Acceptance: launch post + site announcement + release notes ready.

Dependencies:
- Week 3 complete.

Risks:
- Overexpansion beyond launch cut.

Definition of done:
- Launch candidate docs and dashboard experience ready for release.

## 4. Phase-based plan (longer horizon)

| Phase | Objective | Target |
|---|---|---|
| Phase 0 | Product framing and architecture | Completed in Week 1 |
| Phase 1 | Dashboard foundation | Completed in Week 2 |
| Phase 2 | Findings and proof views | Completed in Week 3 |
| Phase 3 | Score and analytics maturity | Starts Week 4 |
| Phase 4 | Managed Control Center operations | Week 4 onward |
| Phase 5 | Landing/docs/conversion UX | Week 4 onward |
| Phase 6 | Hardening, QA, release | Final launch gate |

## 5. Fastest path to compelling launch

1. Ship README + PRD + architecture + roadmap docs.
2. Ship static overview dashboard backed by real triage/history data.
3. Ship findings/proof detail pages with working artifact links.
4. Ship one committed sample dashboard under examples/ for quick evaluation.

## 6. Full vision after MVP

- API mode for external integrations.
- Hosted multi-tenant control center with RBAC.
- Queue-backed orchestration for larger managed fleets.
- Extended alerting and digest automation.

## 7. Release acceptance checklist

- [ ] README and docs pack merged.
- [ ] Schema validation enforced in CI.
- [ ] Dashboard generation deterministic and no-CDN.
- [ ] Finding/proof links resolve via manifest checks.
- [ ] Managed client scoping validated.
- [ ] Claim guardrails documented and reviewed.
